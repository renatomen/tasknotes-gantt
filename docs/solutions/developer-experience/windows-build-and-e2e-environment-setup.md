# Building and running obsidian-gantt (build + E2E) on a fresh Windows machine

## Context

Returning to this repo on a fresh Windows machine (after it had been dormant since Jan 2026), the goal was simply to verify that the build and the WebdriverIO + `wdio-obsidian-service` E2E harness still run. Instead, every step failed in a chain — and each failure masked the next, so the real root causes only became visible one at a time. None of the four root causes were documented anywhere in the repo. This doc captures the whole chain and the verified working setup so the next person (or the same person on another machine) does not re-derive it.

The end state is verified working: `npm install` → `npm run build` (produces `dist/` and installs the plugin into the test vault) → `npm run e2e` (downloads Obsidian, boots it, loads the plugin, smoke spec passes).

## Guidance

The project targets **Node 20** and is driven by a per-machine **`.env`**. On Windows with antivirus SSL scanning, you also need to make Node trust the AV's root cert. The single working command skeleton:

```bash
# 1. Use Node 20 (NOT the system Node 24) — via fnm
export PATH="/c/Users/<you>/AppData/Local/Microsoft/WinGet/Links:$PATH"
eval "$(fnm env --shell bash)"; fnm use 20         # fnm install 20 first, once

# 2. Trust the AV's TLS-scanning root cert (Node 20 lacks --use-system-ca)
export NODE_EXTRA_CA_CERTS="C:/Users/<you>/av-root-ca.pem"

# 3. Mitigate AV-induced download stalls
export NODE_OPTIONS="--dns-result-order=ipv4first"

# 4. Point the vault env at a real local vault (see .env below)
npm install --maxsockets=4 --fetch-timeout=900000 --fetch-retries=8
npm run build      # builds dist/ AND copies plugin into OBSIDIAN_TEST_VAULT
npm run e2e        # override OBSIDIAN_TEST_VAULT to a disposable local copy first
```

The four independent fixes, in the order they were discovered:

### 1. Antivirus SSL/TLS scanning intercepts HTTPS (Norton here)

Norton 360 / Norton Security Ultra "Web/Mail Shield" MITMs all HTTPS locally — confirmed by the cert presented for `registry.npmjs.org` being issued by `CN=Norton Web/Mail Shield Root … generated by Norton Antivirus for SSL/TLS scanning`. This is NOT a network proxy; it happens on a plain home network. It produces two distinct symptoms:

- **Cert-validation failures** because the substitute cert chains to a root only in the Windows trust store, not in tools' bundled CA lists. Per tool:
  - **npm/npx (Node 22+):** `NODE_OPTIONS=--use-system-ca`. **Node 20 does not support that flag** — use `NODE_EXTRA_CA_CERTS=<exported-root.pem>` instead.
  - **cargo:** `CARGO_HTTP_CHECK_REVOKE=false` (disables only OCSP/CRL revocation; chain still validated).
  - **winget:** scope to `--source winget` (the `msstore` source errors `0x8a15005e`).
  - Never disable verification wholesale (`NODE_TLS_REJECT_UNAUTHORIZED=0`, `strict-ssl false`).
- **`npm ERR! code ERR_SOCKET_TIMEOUT`** on large tarballs — the AV scans each download and stalls big native binaries (`@swc/core`, `esbuild`, rollup native) until the socket times out, which aborts the whole install and leaves partially-extracted/corrupt packages (e.g. truncated `.node` → `ERR_DLOPEN_FAILED`). Mitigate by pausing AV web scanning for the install, `--dns-result-order=ipv4first`, lower `--maxsockets`, and high `--fetch-retries`. After a botched install: `npm cache verify`, `rm -rf node_modules`, reinstall.

Export the AV root cert to PEM (PowerShell):

```powershell
$c = Get-ChildItem Cert:\LocalMachine\Root, Cert:\CurrentUser\Root |
     Where-Object { $_.Subject -like '*Norton*' } | Select-Object -First 1
"-----BEGIN CERTIFICATE-----`n" +
  [Convert]::ToBase64String($c.RawData,'InsertLineBreaks') +
  "`n-----END CERTIFICATE-----" | Set-Content "$HOME\norton-ca.pem" -Encoding ascii
```

### 2. Node version: use 20, not the system default

The project (and CI) pins **Node 20**. On **Node 24** the native `classic-level` dependency (pulled in by the wdio stack) has no prebuilt for the Node 24 ABI, so npm falls back to a source compile that fails (`node-gyp` 404 on `win-x86/node.lib`). The system also had a stale global **npm 8** shadowing Node 24's bundled npm. Fix: install Node 20 with a version manager (`fnm install 20`; `nvm` works too), which also gives a matching npm 10. Node 20 uses `classic-level`'s prebuilt — no compilation.

### 3. npm optional-dependencies bug: missing rollup native binary

A clean install can omit the platform-specific optional package `@rollup/rollup-win32-x64-msvc`, so the vite build fails with `Cannot find module '@rollup/rollup-win32-x64-msvc'` ([npm/cli#4828](https://github.com/npm/cli/issues/4828), commonly triggered when `package-lock.json` was generated on Linux CI). Fix:

```bash
npm install --no-save @rollup/rollup-win32-x64-msvc@$(node -p "require('./node_modules/rollup/package.json').version")
```

### 4. The vault path is `.env`-driven, not hardcoded

`scripts/install-to-vault.cjs` (the `postbuild` step) and `test/wdio/wdio.conf.mts` both read `OBSIDIAN_TEST_VAULT` (via dotenv). With no `.env`, they fall back to **another dev machine's hardcoded path** (`C:\Users\renato\...`), so on a fresh machine `npm run build` fails its postbuild with `EPERM: mkdir 'C:\Users\renato'`. The `renato` path is not a typo to fix — it is just a stale default; the correct fix is to create `.env` (gitignored). The **same variable drives both** the manual-install copy and the E2E harness, so:

```ini
# .env (gitignored) — point at the LIVE vault for manual checking
OBSIDIAN_TEST_VAULT=G:/My Drive/.../ObsidianVaults/obsidian-gantt
```

For **E2E**, override it inline to a **disposable local copy** so automated runs never mutate (or sync-pollute) the live vault — especially important when the vault lives on Google Drive / OneDrive:

```bash
OBSIDIAN_TEST_VAULT="C:/Users/<you>/obsidian-test-vaults/obsidian-gantt" npm run e2e
```

(`dotenv` does not override an already-set shell variable, so the inline export wins.)

## Why This Matters

Each blocker individually costs hours, and together they compound into a day-long yak-shave that looks like "the repo is broken" when it is actually four orthogonal environment issues:

- **AV TLS scanning is invisible** — there is no proxy, no VPN, nothing in network settings. The only tell is that `--use-system-ca` "fixes" npm (because the AV root is in the OS store but not Node's bundle). Without naming Norton, you chase phantom "corporate proxy" theories.
- **The Node-version trap is silent** — `package.json` says `engines: >=18`, which is misleading; only Node 20 actually works because of `classic-level`'s prebuilt coverage. Newer is not safer here.
- **The vault fallback fails loudly but misleadingly** — `mkdir 'C:\Users\renato'` looks like a code bug; it is actually missing local config.

Documenting the chain turns a multi-hour rediscovery into a 10-minute checklist.

## When to Apply

- Setting up obsidian-gantt on any new/reimaged Windows machine.
- Any machine where antivirus does HTTPS/SSL scanning (Norton, Kaspersky, ESET, Bitdefender, Avast) — same `NODE_EXTRA_CA_CERTS` approach, different root-cert name.
- When `npm install` shows cert errors or `ERR_SOCKET_TIMEOUT`, a native module fails to build, or `npm run build` can't find a `@rollup/rollup-*` package.

## Examples

| Symptom | Root cause | Fix |
|---|---|---|
| `npm ERR! UNABLE_TO_VERIFY_LEAF_SIGNATURE` / cargo `CRYPT_E_NO_REVOCATION_CHECK` / winget `0x8a15005e` | AV substitutes a TLS cert from a root only in the OS store | `NODE_EXTRA_CA_CERTS` (Node 20) / `CARGO_HTTP_CHECK_REVOKE=false` / `--source winget` |
| `npm ERR! code ERR_SOCKET_TIMEOUT` mid-install | AV scanning stalls large tarballs | pause AV web scan; `--dns-result-order=ipv4first`; low `--maxsockets`; high `--fetch-retries` |

| build: `Cannot find module '@rollup/rollup-win32-x64-msvc'` | npm optional-dep bug | `npm i --no-save @rollup/rollup-win32-x64-msvc@<rollup version>` |
| `postbuild` EPERM `mkdir 'C:\Users\renato'` | no `.env`; fell back to other machine's default vault | create `.env` with `OBSIDIAN_TEST_VAULT` |
